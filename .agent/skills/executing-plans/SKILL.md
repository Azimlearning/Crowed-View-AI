# Executing Plans

## When to use this skill
- A detailed implementation plan has been written and saved (e.g., via the `writing-plans` skill).
- The user asks to execute, build, or implement an approved plan.
- You are transitioning from the planning phase into the execution phase.

## Workflow
- [ ] **1. Load and Review**: Read the plan file and critically review it for potential blockers, missing paths, or unclear instructions.
- [ ] **2. Check for Concerns**: If there are issues, raise them with the user before starting. Otherwise, proceed.
- [ ] **3. Execute Batch**: Execute the first set of tasks (default: up to 3 tasks) following the plan's steps exactly.
- [ ] **4. Run Verifications**: Do not skip tests. Execute specified verification commands and check outputs.
- [ ] **5. Report Progress**: Show what was implemented, the verification output, and wait for user feedback.
- [ ] **6. Iterate**: Apply feedback, execute the next batch, and repeat until the entire plan is complete.

## Instructions

Load the existing architecture/implementation plan, review it critically, and execute tasks in small bite-sized batches. Always stop and report for review between batches. 

**Execution Principles:**
- **Follow Exactly:** Follow every step in the plan precisely as written. Do not guess or skip verification steps.
- **Batching:** Execute up to 3 tasks per batch to ensure frequent checkpoints and allow the user to review progress.
- **Stop on Blockers:** STOP executing immediately and ask the user for help if:
  - You hit a blocker mid-batch (e.g., missing dependency, test consistently fails, unclear instructions).
  - The plan has critical gaps preventing you from starting a task.
  - Verification fails repeatedly after an initial fix attempt.
  - You do not understand an instruction.
- **Never force through blockers.** If something fails, stop and solve it with the user or propose revisiting that section of the plan.
- **Branch/Environment Safety:** Ensure you are working in the correct branch or environment. Do not blindly overwrite production or core files without verifying instructions.

**Reporting Format Between Batches:**
When a batch is complete, output a concise summary:
- **Completed:** List the tasks successfully completed in this batch.
- **Changes:** High-level overview of files touched.
- **Verification:** Outline test and verification results.
- **Next Steps:** Ask *"Are you ready for me to proceed to the next batch, or would you like to review these changes?"*
